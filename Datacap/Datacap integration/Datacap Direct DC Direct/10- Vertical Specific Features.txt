The Check-In transaction utilizes the <PreAuth> or <EMVPreAuth> transaction if the amount is not meant to be settled. However, if the amount should be settled (e.g., if there is an advance deposit or fee to be charged) this can be done with a Sale or EMVSale. Also, if all that is needed at check in is to validate the card, a <ZeroAuth> or <EMVZeroAuth> can be used.Check-outA Check-Out transaction closes out a Check-In transaction and adds the transaction to the current batch. A Check-Out transaction utilizes the <PreAuthCaptureByRecordNo> transaction if the Check-In transaction was a PreAuth or EMVPreAuth and has not yet been added to a batch. Otherwise, a new Sale or EMVSale can be run. The Check-Out amount can be edited with an <AdjustByRecordNo> transaction. The transaction request includes the <AuthCode>, <RefNo>, <AcqRefData> and <RecordNo> from the Check-In transaction.Incremental AuthorizationUse the <IncrementalAuthByRecordNo> transaction to add to the authorized amount on a credit card. Incremental authorization in the lodging industry is typically used for additional duration and extra charges added to a customer’s stay.? Single Stay—To charge for a Single Stay use the Sale or EMVSale transaction. This will authorize the associated amount and add it to the current batch. If a batch does not exist, this transaction creates one.Note: The duration for a Single Stay defaults to one day.? Advance Deposit—To run an Advance Deposit, use the Sale or EMVSale transaction. The merchant must ensure that the transaction amount does not exceed the total price of the reserved services or activity. The cardholder must be informed of the total price of the services or activity, the advance deposit amount, the advance deposit confirmation code, and the cancellation terms.?
