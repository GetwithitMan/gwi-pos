# Stable Client-Generated IDs — Foundational Contract

> Every entity created by a client device MUST use a client-generated UUID as its primary key. The server MUST use that ID. The local event system MUST use the same ID. This is non-negotiable.

---

## Problem This Solves

A critical duplicate-item bug existed across all Android devices (register and PAX). When a user tapped "Add Item":

1. The Android client sent `POST /api/orders/{id}/items` to the NUC server
2. The NUC server created an `OrderItem` with a **server-generated cuid** (e.g., `clx7abc...`)
3. The Android client simultaneously created a local `ITEM_ADDED` event with a **client-generated UUID** (e.g., `550e8400-...`)
4. The local event was projected into a `CachedOrderItem` with the UUID as its `id`
5. When the socket echo arrived from the server, `INSERT OR IGNORE` could not deduplicate because the IDs were different
6. Result: **two items appeared** — one from the server response, one from the local event

This affected every item add on every Android device. The root cause was simple: the server and client generated different IDs for the same entity, so the dedup mechanism (which relies on matching IDs) could never work.

---

## The Contract

1. Every entity created by a client (items, payments, events) MUST have its ID generated client-side
2. The ID is a UUID v4 (`UUID.randomUUID().toString()` on Android)
3. The ID is sent to the server in the request payload (e.g., `lineItemId` field)
4. The server uses the client-provided ID as the entity's primary key (falls back to auto-generated cuid only for backward compatibility with old clients)
5. The local event system uses the same ID (e.g., `ITEM_ADDED` event payload contains the same `lineItemId`)
6. Socket echoes and cross-device sync deduplicate via `INSERT OR IGNORE` on the matching ID

## Data Flow Diagram

```
Client (Android)                    NUC Server                     Neon Cloud
─────────────────                   ──────────                     ──────────
1. User taps "Add Item"
2. Generate lineItemId = UUID v4
3. POST /api/orders/{id}/items
   { lineItemId: "abc-123", ... }
                                    4. Create OrderItem with
                                       id = "abc-123" (client ID)
                                    5. Emit socket event
                                       { lineItemId: "abc-123" }
                                    6. Return 200 OK
7. Create local ITEM_ADDED event
   { lineItemId: "abc-123" }
8. Project to CachedOrderItem
   id = "abc-123"
                                    9. Socket echo arrives
                                       INSERT OR IGNORE → no-op
                                       (id "abc-123" already exists)
                                                                   10. Neon sync replicates
                                                                       OrderItem "abc-123"
```

The critical invariant: steps 3, 5, 7, and 8 all use the **same UUID**. If any step uses a different ID, the dedup in step 9 fails and a duplicate appears.

## Where IDs Are Generated

| Entity | Client Field | Server Field | Generated By | Dedup Mechanism |
|--------|-------------|--------------|--------------|-----------------|
| OrderItem | `OrderItemRequest.lineItemId` | `OrderItem.id` | `UUID.randomUUID()` in `AddItemUseCase` | `INSERT OR IGNORE` on `id` |
| OrderEvent | `OrderEventEntity.eventId` | `OrderEvent.id` | `UUID.randomUUID()` in `OrderMutationRepository` | `INSERT OR IGNORE` on `eventId` |
| Payment | `PaymentApplied.paymentId` | `Payment.id` | Client-generated | `INSERT OR IGNORE` on `id` |

## Rules for Future Development

1. **NEVER** generate an entity ID on the server if the client can provide one
2. **ALWAYS** pass `lineItemId` when constructing `OrderItemRequest`
3. **NEVER** create a local event with a different ID than what was sent to the server
4. The server **MUST** use the client-provided ID if present (fallback to cuid only for old clients that do not send one)
5. Socket echo dedup relies on matching IDs — breaking this contract causes duplicates
6. UUID v4 collision probability: 1 in 5.3x10^36 — safer than any timestamp-based scheme
7. This pattern applies to **ALL** client-created entities, not just order items

---

## Anti-Patterns

| Pattern | Why It Breaks |
|---------|--------------|
| Server generates ID, client generates different ID | **DUPLICATE** — dedup cannot match mismatched IDs |
| Using timestamp-based IDs | Collision risk with multiple devices adding items in the same millisecond |
| Relying on `fetchAndCacheFullOrder` after server add | Race condition — socket echo can arrive before the fetch completes, causing a duplicate |
| Creating local event WITHOUT matching `lineItemId` after server success | **DUPLICATE** — the local projection has one ID, the server has another |
| Omitting `lineItemId` from `OrderItemRequest` | Server falls back to cuid, client uses UUID — same as the original bug |
