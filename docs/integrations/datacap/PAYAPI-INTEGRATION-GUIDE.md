# Datacap PayAPI V2 Integration Guide

## Architecture

GWI POS uses two distinct Datacap integration paths:

| Path | Protocol | Use Case | Client |
|------|----------|----------|--------|
| **Terminal EMV** | XML over TCP to local reader | Card-present (tap/dip/swipe) | `DatacapClient` in `src/lib/datacap/client.ts` |
| **PayAPI V2** | REST over HTTPS to Datacap cloud | Card-not-present (token-based) | `PayApiClient` in `src/lib/datacap/payapi-client.ts` |

PayAPI is used when no physical card reader is involved:
- Online ordering payments
- Membership recurring billing
- Saved card charges (customer links a card via tokenization, future charges use the stored token)
- Text-to-pay deposits (e.g., bakery cake orders)

PayAPI is NOT used for in-person POS payments. Those always go through the terminal EMV path via the local card reader on the NUC's LAN.

## Credential Setup

PayAPI requires an eCommerce Merchant ID (MID) and API Key, separate from the terminal EMV credentials.

### Environment Variables

| Variable | Description | Example |
|----------|-------------|---------|
| `DATACAP_PAYAPI_MID` | eCommerce Merchant ID | `887000XXXXXX` |
| `DATACAP_PAYAPI_KEY` | PayAPI authentication key | `abc123...` |
| `DATACAP_PAYAPI_ENV` | `cert` or `production` | `cert` |

These are configured per venue in Mission Control and injected into the NUC's `.env`. The `PayApiClient` constructor throws if MID or KEY are missing.

## Authentication

PayAPI uses HTTP Basic Auth. The MID and API Key are concatenated with a colon and Base64-encoded:

```
Authorization: Basic base64(MID:API_KEY)
```

Implementation in `PayApiClient.buildAuth()`:
```typescript
const credentials = Buffer.from(`${this.mid}:${this.apiKey}`).toString('base64')
return `Basic ${credentials}`
```

Every request also includes:
- `User-Agent: GWI-POS/1.0.0`
- `Accept: application/json`
- `Content-Type: application/json` (for POST/PUT)

## Token Lifecycle

PayAPI operates on tokens, not raw card numbers (PCI compliance). Three token types:

### One-Time Use (OTU)

- Generated by Datacap's hosted tokenization page or iframe
- Valid for 15 minutes
- Single use: consumed on the first transaction
- Used for: online ordering checkout, text-to-pay

### Multi-Use (Stored Card)

- Returned in the `token` field of a successful Sale or PreAuth response
- Can be stored in the database for future charges
- Used for: saved cards on customer profiles, "charge same card again" workflows
- Stored on `CardProfile.token` in the POS schema

### Recurring

- Requires `RecurringData` chain from the initial authorization
- Each subsequent charge must include the `RecurringData` from the previous response
- Used for: membership billing cycles
- Chain preservation is critical -- if `RecurringData` is lost, the chain breaks and the processor may decline future charges

**Chain corruption warning:** The client logs a warning if `RecurringData` was sent in a request but missing from the response. The caller should preserve the old chain value in this case.

## Operations

### Sale

Charge a card immediately using a token.

```
POST /v2/credit/sale
```

| Field | Required | Description |
|-------|----------|-------------|
| `Token` | Yes | OTU or multi-use token |
| `Amount` | Yes | Charge amount, 2 decimal places (e.g., "15.11") |
| `InvoiceNo` | Yes | Unique transaction ID (use order number) |
| `Tip` | No | Tip amount (e.g., "2.50") |
| `Tax` | No | Tax amount (e.g., "1.25") |
| `CVV` | No | From tokenization response |
| `RecurringData` | No | Chain data for recurring billing |

### Void

Cancel a completed sale before batch settlement.

```
POST /v2/credit/sale/{RefNo}/void
```

| Field | Required | Description |
|-------|----------|-------------|
| `Token` | Yes | Token from original sale |
| `InvoiceNo` | Yes | Original invoice number |

The `RefNo` from the original sale response goes in the URL path.

### Refund (Blind Return)

Return funds to a card without referencing the original sale. Used when the original `RefNo` is unavailable or the batch has already settled.

```
POST /v2/credit/return
```

| Field | Required | Description |
|-------|----------|-------------|
| `Token` | Yes | Token for the card to refund |
| `Amount` | Yes | Refund amount |
| `InvoiceNo` | Yes | New unique invoice number |

### Pre-Authorization

Hold funds without capturing. Used for deposits or holds.

```
POST /v2/credit/preauth
```

| Field | Required | Description |
|-------|----------|-------------|
| `Token` | Yes | OTU or multi-use token |
| `Amount` | Yes | Hold amount |
| `InvoiceNo` | Yes | Unique transaction ID |
| `CVV` | No | From tokenization response |

### Capture

Finalize a pre-authorized hold with the actual charge amount.

```
PUT /v2/credit/preauth/{RefNo}
```

| Field | Required | Description |
|-------|----------|-------------|
| `Token` | Yes | Token from pre-auth response |
| `Amount` | Yes | Final capture amount (can differ from pre-auth) |
| `InvoiceNo` | Yes | Original invoice number |
| `Tip` | No | Tip amount |

### Void Pre-Auth

Cancel a pre-authorization before capture.

```
POST /v2/credit/preauth/{RefNo}/void
```

| Field | Required | Description |
|-------|----------|-------------|
| `Token` | Yes | Token from pre-auth response |
| `InvoiceNo` | Yes | Original invoice number |

### Transaction Lookup

Retrieve details of a transaction by reference number.

```
GET /v2/credit/{RefNo}
```

No body required. Returns the full transaction record.

## RecurringData Chain Preservation

For membership billing, the `RecurringData` field creates a chain linking transactions:

```
Initial Sale (with OTU token)
  -> Response includes RecurringData: "abc123..."
     Store this value

Month 2 Sale (with multi-use token + RecurringData from Month 1)
  -> Response includes RecurringData: "def456..."
     Update stored value

Month 3 Sale (with multi-use token + RecurringData from Month 2)
  -> ...and so on
```

If the chain is broken (RecurringData not sent or lost), the processor may:
- Decline the transaction
- Flag it as a new authorization instead of recurring
- Lose chargeback protection afforded to recurring merchants

The `PayApiClient` passes `RecurringData` through on Sale requests and logs a warning if it goes missing from responses.

## Error Handling

### HTTP Status Codes

| Code | Meaning |
|------|---------|
| 200 | Approved / Success |
| 400 | Bad request (missing fields, invalid token) |
| 401 | Authentication failure (bad MID/Key) |
| 402 | Declined by processor |
| 404 | RefNo not found |
| 408 | Timeout (client-side, 5s circuit breaker) |

### Response Status Values

| Status | Meaning |
|--------|---------|
| `Approved` | Transaction approved by processor |
| `Success` | Administrative operation succeeded (void, lookup) |
| `Declined` | Processor declined the transaction |
| `Error` | System or configuration error |

Use the helper `isPayApiSuccess(status)` to check for either `Approved` or `Success`.

### Timeout and Retry

The client enforces a 5-second circuit breaker (`PAYAPI_TIMEOUT_MS = 5000`). If the processor does not respond within 5 seconds, the request is aborted and a `PayApiError` with status 408 is thrown.

**Retry policy:** PayAPI requests are NOT automatically retried. The caller is responsible for retry logic because:
- Sales and captures are not idempotent without deduplication
- A timed-out request may have succeeded on the processor
- The `InvoiceNo` field can be used for deduplication on retry

### PCI Safety

Error messages never include token values or account numbers. The `PayApiError` message is sanitized:
```
PayAPI error: {message} (HTTP {status}, ReturnCode {code})
```

## Singleton Access

The `PayApiClient` is accessed via a lazy singleton:

```typescript
import { getPayApiClient } from '@/lib/datacap/payapi-client'

const client = getPayApiClient()
const result = await client.sale({ token, amount: '15.11', invoiceNo: 'ORD-123' })
```

The singleton is constructed on first call and reads env vars at that time. Throws immediately if MID or KEY are missing.

## Testing with Cert Environment

1. Set `DATACAP_PAYAPI_ENV=cert` (this is the default)
2. Use Datacap-provided test MID and API Key
3. Use Datacap's hosted tokenization test page to generate OTU tokens
4. Test card numbers from Datacap's certification guide (Visa 4111..., MC 5111..., etc.)
5. Verify `status: 'Approved'` in responses
6. Check `returnCode` matches expected values from Datacap's cert matrix

**Note:** Cert environment does not process real money. Batch close in cert mode does not settle funds.
