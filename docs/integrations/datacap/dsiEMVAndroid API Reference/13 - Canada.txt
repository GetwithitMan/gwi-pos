Canada
Overview
dsiEMVAndroid is supported to Global Canada using the PAX A920Pro or PAX IM30 devices. It is also supported to WorldPay using the PAX A800. Canadian transaction processing was added to dsiEMVAndroid version 1.52 for PAX Android devices.
Differences between US and Canadian Integration
1. Support for Credit and Debit(Interac) is required.
2. Store and Forward (offline processing) is not supported in Canada
3. Cash Back is not allowed
4. SurchargeWithLookup is not supported
5. PaymentFee is not supported
6. EMVSaleWithSplit is not supported
7. SequenceNo value is returned by the processors in Canada. The SequenceNo value is tracked per terminal
8. Prompting for tip at time of sale is supported
9. Support for English and French languages
10. Additional XML Response tags are returned for receipt printing specific to merchant and customer language.
11. Receipt data is returned for merchant and customer copy
Tag
	Description
	<ReceiptLanguage>
	(String 6-7 chars) Returns a value of French or English. This value is used by the POS to determine the language to be used to print the Merchant receipt header (“TRANSACTION RECORD” or “RELEVE DE TRANSACTION”)
	<CustomerReceiptLanguage>
	(String 6-7 chars) Returns a value of French or English. This value is used by the POS to determine the language to be used to print the Customer receipt header (“TRANSACTION RECORD” or “RELEVE DE TRANSACTION”)
	<ISORespCode>
	(Number 3 chars) 3 digit response code from Canadian Host.
	Receipts
For Canada, receipt data is returned for the merchant and customer copy. Two separate sets of receipt data are returned. The merchant receipt is returned in tags <Line1> to <LineN>. The customer receipt is returned in tags <CustLine1> to <CustLineN>. The merchant receipt is generated in the language of the merchant and corresponds with Merchant Language set as the PSCS parameter. The customer receipt is generated in the cardholder language. Before printing the receipt, the POS should still print the receipt header, merchant name, address, and phone number as separate lines at the top of the receipt. Receipt examples are available in EMVTransactions section.
EMV Transactions and Operations Supported by dsiEMVAndroid Canada
dsiEMVAndroid supports the following transactions:
Credit/Debit Transactions
	Definition
	Supported by
	EMVPadReset
	This command must be performed after every monetary transaction (Sale, VoidSale, Return, VoidReturn, VoiceAuth, ZeroAuth, etc) to assure that no card remains in the PIN pad chip reader. If no card is in the reader, then a response should be returned in 2-3 seconds. If there’s a card in the reader, the PIN pad displays ‘Remove Card’ and waits for the card to be removed before returning a response.
	Credit or Debit
	EMVSale
	EMV Sale transaction.
	Credit or Debit
	EMVVoidSale
	Void a previously completed sale processed as Credit or Interac Debit. Voiding a sale processed as debit is only supported in Canada
	Credit or Debit
	VoidSaleByRecordNo
	Void of a previously completed EMV Sale transaction using transaction reference identifier(s).
	Credit
	SaleByRecordNo
	Process an EMV Sale transaction without a card present.
	Credit
	EMVPreAuth
	Preauthorize a credit card which must be subsequently captured for settlement.
	Credit
	PreAuthCaptureByRecordNo
	To capture previously processed EMVPreAuth transaction and add to the current batch.
	Credit
	EMVReturn
	EMV Return transaction.
	Credit or Debit
	EMVVoidReturn
	Void a previously completed Return processed as Credit or Interac Debit. Voiding a return processed as debit is only supported in Canada
	Credit or Debit
	ReturnByRecordNo
	Process an EMV Return transaction without a card present.
	Credit
	VoidReturnByRecordNo
	Void of a previously completed EMV Return transaction using transaction reference identifier(s).
	Credit
	EMVZeroAuth
	Verify a card is valid without charging any amount.
	Credit
	

PrePaid Transactions
	Definition
	PrePaid EMVIssue
	Issue a new prepaid (stored value) card with a starting balance (value).
	PrePaid EMVVoidIssue
	Void the previous issue a new prepaid (stored value) card with a starting balance (value).
	PrePaid EMVSale
	Process a payment (purchase) using a prepaid card. PrePaid sales reduce the available balance (value) of the account (card).
	PrePaid EMVVoidSale
	Void a previous a payment (purchase) on a prepaid card. PrePaid void sales increase the available balance (value) of the account (card).
	PrePaid EMVReturn
	Process a return (refund) using a prepaid card. PrePaid returns increase the available balance (value) of the account (card).
	PrePaid EMVVoidReturn
	Void a previous a return (refund) on a prepaid card. PrePaid void returns decrease the available balance (value) of the account (card).
	PrePaid EMVCashOut
	Process a cash refund of the remaining balance on a prepaid card. A CashOut reduces the available balance (value) of the account (card) to zero.
	PrePaid EMVBalance
	Find the remaining balance (value) on a prepaid card.
	

Admin Transactions
	Definition
	EMVParamDownload
	To download required parameters and security information to the EMV device.
	BatchSummary

	Canada EMV Transactions
EMVSale
To process a payment using an EMV device. Credit and Debit transactions are supported presented as chip, contactless, MSR, and manual entry.
Implementation Notes:
1. To request a token, set <RecordNo> to RecordNumberRequested and <Frequency> to Recurring
2. Manual card account entry with the <AcctNo> tag should only be used if the PIN pad insertion reader fails to get card account information on the first transaction attempt. The POS logic should allow the transaction to be run again with the AcctNo tag included to force a manual entry of card account information.

MerchantID
(String 1-24 chars)
	Required
Merchant identification assigned by gateway or processor.
	POSPackageID
(String 1-99 chars)
	Required
Include this tag with POS package name and version where the name and version are separated with a colon (“:”). Ex: EMVUSClient:1.26
	SecureDevice
(String 1-40 chars)
	Required
Secure Device type. See Supported Devices for ValidSecureDeviceID values.
	TranCode
(String 1-40 chars)
	Required
The type of transaction (eg. sale, return ,void, etc)
	InvoiceNo
(Number 1-8 chars)
	Required
Invoice number - sequential receipt number, check number, or other unique transaction identifier created and supplied by POS system.
	RefNo
(String 1-50 chars)
	Required
A reference returned by the processor. Should initially match InvoiceNo. The RefNo value returned by the processor should be saved since it is required for subsequent VoidSale, Adjust, VoiceAuth and PreAuthCapture transactions.
	Amount:Purchase
(Number 1-8 chars)
	Required
Purchase price (with 2 place decimal – e.g. 29.95)
	SequenceNo
(String 1-20 chars)
	Required
Use the value returned for SequenceNo from the last response. If the SequenceNo is lost or for initial deployment of the PIN pad, the ECR/POS should attempt any transaction (using “0010010010” as a SequenceNo value) to re-sync.
	BluetoothDeviceName
(String 1-24 chars)
	Conditional
Required for Bluetooth communication. Identifier returned by EMV device during BLE discovery.
	OperationMode
(String 1-4 chars)
	Conditional
used to direct transactions to CERT or PROD NETePay Hosted environment. Supported values are “CERT” and “PROD”. Default value is “PROD”.
	PinPadIpAddress
(String 7-15 chars)
	Conditional
IP address of EMV device on local network.
	PinPadIpPort
(Number 1-5)
	Conditional
IP Port of EMV device. For PAX devices, default port is 1235.
	RecordNo
(String 1-256 chars)
	Conditional
Required to obtain a Datacap token in response for use in subsequent transactions. Include with value “RecordNumberRequested”. Omit this tag if a token is not required.
	Frequency
(String 1-40 chars)
	Conditional
Required when <RecordNo> is included to obtain a token. Supported values are “OneTime” or “Recurring”. To request a token that can be used more than once, use the value “Recurring”. Omit this tag if <RecordNo> tag is not supplied.
	PartialAuth
(String 1-20 chars)
	Use “Allow” to allow acceptance of partial authorizations.
	OperatorID
(String 1-24 chars)
	Operator (clerk, server, etc.) associated with the transaction.
	UserTrace
(String 0-40 chars)
	Echo of data supplied by the user system in the request; for use by the user system for internal tracking. May be null if no user system data was supplied in the request
	CardType
(String 3-24 chars)
	“Credit” OR “Debit”. When an EMV card is presented, this will ensure only Credit or Debit processes (the user may still be presented with EMV Credit and Debit AIDs). For a swiped transaction, this field instructs the EMV device to select the transaction type without giving the cardholder to option to select at the PIN pad.
	CollectData
(String 1-40 chars)
	“CardholderName” – To optionally extract the cardholder name from the card. If name is available, it will be returned in the <CardholderName> tag in the response and the name will be included in the draft print output.
	Account:AcctNo
(String 1-24 chars)
	valid values are “SecureDevice” or “Prompt”. “Prompt” will prompt on EMV device to manually key card information.
	ReturnClearExpDate
(String 1-5 chars)
	Use “Allow” to return unmasked card expiraton date in <ExpDateMonth> and <ExpDateYear>.
	Amount:Gratuity
(Number 1-8 chars)
	Use “Prompt” to prompt cardholder to input gratuity amount at time of sale.
	Amount:Tax
(Number 1-8 chars)
	For Purchase Card Level II transactions– Sales Tax amount (with 2 decimal places – eg. 29.95). The Tax amount is included in the purchase amount.
	TranInfo:CustomerCode
(String 1-17 chars)
	For Purchase Card Level II transactions– customer’s identifying information; such as PO number.
	ProcessorToken
(String 10-14 chars)
	include value “TokenRequested” to receive processor specific token in response. ProcessorToken is only supported to Chase Paymentech.
	LaneID
(Number 1-8 chars)
	A unique LaneID (lane number) can be supplied to identify each POS/ECR in a single store location with multiple payment points
	OKAmount
(String 1-24 chars)
	When the optional OKAmount tag is included with the value “Disallow”, the PIN pad will not prompt the cardholder to approve the transaction amount. When the optional OKAmount tag is included with the value “Disallow_All”, the pinpad will not prompt the cardholder to approve the transaction amount or any added gratuity or cashback amounts after selection. If the OKAmount tag is omitted, the PIN pad will ask the cardholder to approve the transaction amount by default.

ResponseOrigin
(String 1-10 chars)
	Indicates the source of the response:
“Client” = generated by dsiEMVAndroid application
“Server” = generated by Datacap server
“Processor” = generated by payment processor.
	DSIXReturnCode
(Number 6-6 chars)
	Six digit return code that identifies the error type. See Errors on error codes for possible values.
	CmdStatus
(String 1-10 chars)
	Indicates the outcome of the command:
“Success” = command completed successfully
“Approved” = transaction approved by payment processor
“Declined” = transaction declined by payment processor
“Declined Offline” = chip transaction transaction declined offline
“Error” = error processing command. Check DSIXReturnCode and TextResponse for additional info on error.
	TextResponse
(String 1-255 chars)
	For Successful responses, this field can contain an optional message from the server or processor. For other response codes, see Errors on Error Codes for values.
	SequenceNo
(Number 1-20 chars)
	The sequence number returned in all responses to be used on the next request. If the SequenceNo is lost after initial deployment, attempt any transaction using value “0010010010” to re-sync.
	UserTrace
(String 0-40 chars)
	Echo of data supplied by the user system in the request; for use by the user system for internal tracking. May be null if no user system data was supplied in the request
	MerchantID
(String 1-24 chars)
	The merchant ID supplied in the transaction request.
	AcctNo
(String 1-24 chars)
	The account number from the card used for a manually entered transaction – will be truncated.
	CardType
(String 3-24 chars)
	Card type used on transaction. Values include: AMEX, VISA, M/C, DCLB, DCVR, JCB, Debit, OTHER, UNDETERMINED (returned on tokenized transactions).
	TranCode
(String 1-40 chars)
	TranCode value supplied in the transaction request.
	AuthCode
(String 1-24 chars)
	The authorization code returned by the processor for the approved transaction. The AuthCode value should be saved since it is required for VoidSale, Adjust, VoiceAuth and PreAuthCapture transactions.
	CaptureStatus
(String 1-24 chars)
	Indicates whether the host has captured the transaction. eg. “Captured”, “NotCaptured”
	RefNo
(String 1-50 chars)
	A reference returned by the processor. Should initially match InvoiceNo. The RefNo value returned by the processor should be saved since it is required for subsequent VoidSale, Adjust, VoiceAuth and PreAuthCapture transactions.
	InvoiceNo
(Number 1-8 chars)
	The invoice number supplied in the transaction request.
	OperatorID
(Number 1-10 chars)
	Operator (clerk, server, etc.) associated with the Transaction.
	Purchase
(Number 1-8 chars)
	The purchase amount requested in the transaction (to 2 decimal places).
	Gratuity
(Number 1-8 chars)
	The gratuity amount requested in the transaction.
	Authorize
(Number 1-8 chars)
	The amount actually authorized for the transaction. This amount could be less than the Purchase amount requested and the POS/ECR system should verify to determine if additional tendering is required to satisfy the total payment.
	AcqRefData
(String 1-200 chars)
	Acquirer Reference Data returned by processor.
	ReceiptLanguage
(String 6-7 chars)
	Returns a value of French or English. This value is used by the POS to determine the language to be used to print the Merchant receipt header (“TRANSACTION RECORD” or “RELEVE DE TRANSACTION”) <ReceiptLanguage> is specific to Canadian processing only.
	CustomerReceiptLanguage
(String 6-7 chars)
	Returns a value of French or English. This value is used by the POS to determine the language to be used to print the Customer receipt header (“TRANSACTION RECORD” or “RELEVE DE TRANSACTION”) <CustomerReceiptLanguage> is specific to Canadian processing only.
	EntryMethod
(String 1-40 chars)
	Values returned in French or English. Possible values can include: CHIP, PUCE, MANUAL, and FALLBACK. For contactless, values can include PROXIMITY or SANS CONTACT.
	CVM

	Transaction CVM.
	RecordNo

ResponseOrigin
(String 1-10 chars)
	Indicates the source of the response:
“Client” = generated by dsiEMVAndroid application
“Server” = generated by Datacap server
“Processor” = generated by payment processor.
	DSIXReturnCode
(Number 6-6 chars)
	Six digit return code that identifies the error type. See Errors on error codes for possible values.
	CmdStatus
(String 1-10 chars)
	Indicates the outcome of the command:
“Success” = command completed successfully
“Approved” = transaction approved by payment processor
“Declined” = transaction declined by payment processor
“Declined Offline” = chip transaction transaction declined offline
“Error” = error processing command. Check DSIXReturnCode and TextResponse for additional info on error.
	TextResponse
(String 1-255 chars)
	For Successful responses, this field can contain an optional message from the server or processor. For other response codes, see Errors on Error Codes for values.
	SequenceNo
(Number 1-20 chars)
	The sequence number returned in all responses to be used on the next request. If the SequenceNo is lost after initial deployment, attempt any transaction using value “0010010010” to re-sync.
	UserTrace
(String 0-40 chars)
	Echo of data supplied by the user system in the request; for use by the user system for internal tracking. May be null if no user system data was supplied in the request
	MerchantID
(String 1-24 chars)
	The merchant ID supplied in the transaction request.
	AcctNo
(String 1-24 chars)
	The account number from the card used for a manually entered transaction – will be truncated.
	CardType
(String 3-24 chars)
	Card type used on transaction. Values include: AMEX, VISA, M/C, DCLB, DCVR, JCB, Debit, OTHER, UNDETERMINED (returned on tokenized transactions).
	TranCode
(String 1-40 chars)
	TranCode value supplied in the transaction request.
	AuthCode
(String 1-24 chars)
	The authorization code returned by the processor for the approved transaction. The AuthCode value should be saved since it is required for VoidSale, Adjust, VoiceAuth and PreAuthCapture transactions.
	CaptureStatus
(String 1-24 chars)
	Indicates whether the host has captured the transaction. eg. “Captured”, “NotCaptured”
	RefNo
(String 1-50 chars)
	A reference returned by the processor. Should initially match InvoiceNo. The RefNo value returned by the processor should be saved since it is required for subsequent VoidSale, Adjust, VoiceAuth and PreAuthCapture transactions.
	InvoiceNo
(Number 1-8 chars)
	The invoice number supplied in the transaction request.
	OperatorID
(Number 1-10 chars)
	Operator (clerk, server, etc.) associated with the Transaction.
	Purchase
(Number 1-8 chars)
	The purchase amount requested in the transaction (to 2 decimal places).
	Authorize
(Number 1-8 chars)
	The amount actually authorized for the transaction. This amount could be less than the Purchase amount requested and the POS/ECR system should verify to determine if additional tendering is required to satisfy the total payment.
	AcqRefData
(String 1-200 chars)
	Acquirer Reference Data returned by processor.
	ReceiptLanguage
(String 6-7 chars)
	Returns a value of French or English. This value is used by the POS to determine the language to be used to print the Merchant receipt header (“TRANSACTION RECORD” or “RELEVE DE TRANSACTION”) <ReceiptLanguage> is specific to Canadian processing only.
	CustomerReceiptLanguage
(String 6-7 chars)
	Returns a value of French or English. This value is used by the POS to determine the language to be used to print the Customer receipt header (“TRANSACTION RECORD” or “RELEVE DE TRANSACTION”) <CustomerReceiptLanguage> is specific to Canadian processing only.
	EntryMethod
(String 1-40 chars)
	Values returned in French or English. Possible values can include: CHIP, PUCE, MANUAL, and FALLBACK. For contactless, values can include PROXIMITY or SANS CONTACT.
	CVM
(String 1-40 chars)
	Transaction CVM.
	RecordNo
(String 1-256 chars)
	Returned when optional<RecordNo> tag used in a request.
The RecordNo value can be saved and used as a token to process subsequent ByRecordNo transactions without using account number data.
	CardholderName
(String 1-40 chars)
	Cardholder name returned which may be present if the <CollectData> tag is used in the request.
	PayAPI_Id
(String 1-34 chars)
	The Pay API reference number to be used through Pay API for follow up transactions.
	ExpDateMonth
(Number 2-2 chars)
	Expiration date month in MM format. Returned when optional <ReturnClearExpDate> used in a request.
	ExpDateYear
(Number 2-2 chars)
	Expiration date year in YY format. Returned when optional <ReturnClearExpDate> used in a request.
	ProcessorToken
(String 10-14 chars)
	When included in request, the processor specific token returned in response. ProcessorToken is only supported to Chase Paymentech
	ProcessData
(String 1-999 chars)
	optionally returned by processor. This value must be retained for use in a subsequent requests.
	Date
(String 1-40 chars)
	Transaction date
	Time
(String 1-40 chars)
	Transaction time
	ApplicationLabel
(String 1-40 chars)
	Transaction application label
	AID
(String 1-40 chars)
	Transaction AID
	ISO
(String 1-40 chars)
	Transaction ISO
	TVR
(String 1-40 chars)
	Transaction TVR
	TSI
(String 1-40 chars)
	Transaction TSI

(String 1-41 chars)
	Receipt print lines. Each <LineN> tag should be printed on a new line. Each <LineN> tag begins with a “.” which should not be printed – empty lines (blank) have only “.” as data. When <PrintData> elements exist, they should always be printed as the draft for the transaction.

2. Use the <RefNo>, <AuthCode>, <RecordNo>, <InvoiceNo> ,<ProcessData> and <AcqRefData> values obtained in the response to a previous transaction.
3. Each PreAuthCaptureByRecordNo performed will add a transaction to the current batch. Multiple PreAuthCaptureByRecordNo requests for the same EMVPreAuth will result in duplicate charges to the customer. If a mistake is made, a VoidSaleByRecordNo must be used to Void the transaction.

MerchantID
(String 1-24 chars)
	Required
Merchant identification assigned by gateway or processor.
	POSPackageID
(String 1-99 chars)
	Required
Include this tag with POS package name and version where the name and version are separated with a colon (“:”). Ex: EMVUSClient:1.26
	SecureDevice
(String 1-40 chars)
	Required
Secure Device type. See Supported Devices for ValidSecureDeviceID values.
	TranType
(String 1-40 chars)
	Required
value set “Credit”. Used with the TranCode to specify payment type.
	TranCode
(String 1-40 chars)
	Required
The type of transaction (eg. sale, return ,void, etc)
	InvoiceNo
(Number 1-8 chars)
	Required
Invoice number - sequential receipt number, check number, or other unique transaction identifier created and supplied by POS system.
	RefNo
(String 1-50 chars)
	Required
A reference returned by the processor. Should initially match InvoiceNo. The RefNo value returned by the processor should be saved since it is required for subsequent VoidSale, Adjust, VoiceAuth and PreAuthCapture transactions.
	AuthCode
(String 1-24 chars)
	Required
The authorization code returned by the processor for the approved transaction. The AuthCode value should be saved since it is required for VoidSale, Adjust, VoiceAuth and PreAuthCapture transactions.
	Amount:Purchase
(Number 1-8 chars)
	Required
Purchase price (with 2 place decimal – e.g. 29.95)
	RecordNo
(String 1-256 chars)
	Required
Token returned from previous authorization response.
	Frequency
(String 1-40 chars)
	Required
Required when <RecordNo> is included. Supported values are “OneTime” or “Recurring”.
	SequenceNo
(String 1-20 chars)
	Required
Use the value returned for SequenceNo from the last response. If the SequenceNo is lost or for initial deployment of the PIN pad, the ECR/POS should attempt any transaction (using “0010010010” as a SequenceNo value) to re-sync.
	AcqRefData
(String 1-200 chars)
	Conditional
Required if Acquirer Reference Data is returned by processor in authorization response.
	BluetoothDeviceName
(String 1-24 chars)
	Conditional
Required for Bluetooth communication. Identifier returned by EMV device during BLE discovery.
	OperationMode
(String 1-4 chars)
	Conditional
used to direct transactions to CERT or PROD NETePay Hosted environment. Supported values are “CERT” and “PROD”. Default value is “PROD”.
	PinPadIpAddress
(String 7-15 chars)
	Conditional
IP address of EMV device on local network.
	PinPadIpPort
(Number 1-5)
	Conditional
IP Port of EMV device. For PAX devices, default port is 1235.
	OperatorID
(String 1-24 chars)
	Operator (clerk, server, etc.) associated with the transaction.
	UserTrace
(String 0-40 chars)
	Echo of data supplied by the user system in the request; for use by the user system for internal tracking. May be null if no user system data was supplied in the request
	LaneID
(Number 1-8 chars)
	A unique LaneID (lane number) can be supplied to identify each POS/ECR in a single store location with multiple payment points.

ResponseOrigin
(String 1-10 chars)
	Indicates the source of the response:
“Client” = generated by dsiEMVAndroid application
“Server” = generated by Datacap server
“Processor” = generated by payment processor.
	DSIXReturnCode
(Number 6-6 chars)
	Six digit return code that identifies the error type. See Errors on error codes for possible values.
	CmdStatus
(String 1-10 chars)
	Indicates the outcome of the command:
“Success” = command completed successfully
“Approved” = transaction approved by payment processor
“Declined” = transaction declined by payment processor
“Declined Offline” = chip transaction transaction declined offline
“Error” = error processing command. Check DSIXReturnCode and TextResponse for additional info on error.
	TextResponse
(String 1-255 chars)
	For Successful responses, this field can contain an optional message from the server or processor. For other response codes, see Errors on Error Codes for values.
	SequenceNo
(Number 1-20 chars)
	The sequence number returned in all responses to be used on the next request. If the SequenceNo is lost after initial deployment, attempt any transaction using value “0010010010” to re-sync.
	UserTrace
(String 0-40 chars)
	Echo of data supplied by the user system in the request; for use by the user system for internal tracking. May be null if no user system data was supplied in the request
	MerchantID
(String 1-24 chars)
	The merchant ID supplied in the transaction request.
	AcctNo
(String 1-24 chars)
	The account number from the card used for a manually entered transaction – will be truncated.
	ExpDate
(Number 4-4 chars)
	Expiration date as MMYY
	CardType
(String 3-24 chars)
	Card type used on transaction. Values include: AMEX, VISA, M/C, DCLB, DCVR, JCB, Debit, OTHER, UNDETERMINED (returned on tokenized transactions).
	TranCode
(String 1-40 chars)
	TranCode value supplied in the transaction request.
	AuthCode
(String 1-24 chars)
	The authorization code returned by the processor for the approved transaction. The AuthCode value should be saved since it is required for VoidSale, Adjust, VoiceAuth and PreAuthCapture transactions.
	CaptureStatus
(String 1-24 chars)
	Indicates whether the host has captured the transaction. eg. “Captured”, “NotCaptured”
	RefNo
(String 1-50 chars)
	A reference returned by the processor. Should initially match InvoiceNo. The RefNo value returned by the processor should be saved since it is required for subsequent VoidSale, Adjust, VoiceAuth and PreAuthCapture transactions.
	InvoiceNo
(Number 1-8 chars)
	The invoice number supplied in the transaction request.
	OperatorID
(Number 1-10 chars)
	Operator (clerk, server, etc.) associated with the Transaction.
	Purchase
(Number 1-8 chars)
	The purchase amount requested in the transaction (to 2 decimal places).
	Authorize
(Number 1-8 chars)
	The amount actually authorized for the transaction. This amount could be less than the Purchase amount requested and the POS/ECR system should verify to determine if additional tendering is required to satisfy the total payment.
	AcqRefData
(String 1-200 chars)
	Acquirer Reference Data returned by processor.
	RecordNo
(String 1-256 chars)
	Returned when optional<RecordNo> tag used in a request.
The RecordNo value can be saved and used as a token to process subsequent ByRecordNo transactions without using account number data.
	ISORespCode
(Number 3 chars)
	3 digit response code from Canadian Host. <ISORespCode> is specific to Canadian processing only.
	ReceiptLanguage
(String 6-7 chars)
	Returns a value of French or English. This value is used by the POS to determine the language to be used to print the Merchant receipt header (“TRANSACTION RECORD” or “RELEVE DE TRANSACTION”) <ReceiptLanguage> is specific to Canadian processing only.
	CustomerReceiptLanguage
(String 6-7 chars)
	Returns a value of French or English. This value is used by the POS to determine the language to be used to print the Customer receipt header (“TRANSACTION RECORD” or “RELEVE DE TRANSACTION”) <CustomerReceiptLanguage> is specific to Canadian processing only.
	EntryMethod
(String 1-40 chars)
	Values returned in French or English. Possible values can include: CHIP, PUCE, MANUAL, and FALLBACK. For contactless, values can include PROXIMITY or SANS CONTACT.
	CVM
(String 1-40 chars)
	Transaction CVM.
	PayAPI_Id
(String 1-34 chars)
	The Pay API reference number to be used through Pay API for follow up transactions.
	ExpDateMonth
(Number 2-2 chars)
	Expiration date month in MM format. Returned when optional <ReturnClearExpDate> used in a request.
	ExpDateYear
(Number 2-2 chars)
	Expiration date year in YY format. Returned when optional <ReturnClearExpDate> used in a request.
	ProcessData
(String 1-999 chars)
	optionally returned by processor. This value must be retained for use in a subsequent requests.
	Date
(String 1-40 chars)
	Transaction date
	Time
(String 1-40 chars)
	Transaction time
	ApplicationLabel
(String 1-40 chars)
	Transaction application label
	AID
(String 1-40 chars)
	Transaction AID
	ISO
(String 1-40 chars)
	Transaction ISO
	TVR
(String 1-40 chars)
	Transaction TVR
	TSI
(String 1-40 chars)
	Transaction TSI
	PrintData:LineNNN
(String 1-41 chars)
	Receipt print lines. Each <LineN> tag should be printed on a new line. Each <LineN> tag begins with a “.” which should not be printed – empty lines (blank) have only “.” as data. When <PrintData> elements exist, they should always be printed as the draft for the transaction.

	PrintReceipt
To print a receipt from an EMV device with a built-in printer.
Implementation Notes:
1. Currently available on the PAX A920Pro.
2. Each <LineN> should be preceded with a period character to preserve whitespace. Include a line containing only a period to print a blank line.
3. Responds with <CmdStatus> Success if receipt successfully prints. Responds with <CmdStatus> Error if receipt fails to print.
4. The Merchant and Customer receipt data is returned in the transaction response. Canadian merchants have specific receipt requirements and Datacap’s receipt data should be used for receipt printing to avoid any issues in production.

	GetSignature
To obtain a digitized signature from the EMV device.
View Demo 
Implementation Notes:
1. GetSignature is a separate request that could be sent at anytime to obtain a digitized signature from the customer.
2. An EMVPadReset must be performed prior to GetSignature to ensure the customer’s card is removed from the EMV device.
3. A GetSignature request allows the integrator to optionally obtain a digitized signature from the customer for paperless record keeping of receipts when using a device that support signature capture. When a response contains the <CVM> tag, the values can be one of the following:
“SIGN”
“PIN VERIFIED”
“PIN VERIFIED AND SIGN”
“NONE”

If the response value for <CVM> contains either “SIGN” or “PIN VERIFIED AND SIGN”, then send the GetSignature request. The GetSignature response should be saved with an electronic copy of the receipt for that transaction to support a merchant in a charge back request. The GetSignature response data could be converted to a graphic format, such as .bmp) and included in the stored receipt image. In the absence of a CVM (when a non-EMV card is presented), a signature can still be collected except when CardType is DEBIT.
4. The Signature response data is a string of up to 8192 characters and is returned as a series of “strokes” and within each stroke is a series of X,Y coordinate points. The first point of a stroke is always a “move to” position. All subsequent points within a stroke are “draw to” positions. The “end stroke point” is simply “#,#”. That would be followed by the first point of the next stroke. X and Y coordinates are separated by a comma and end with a colon.
