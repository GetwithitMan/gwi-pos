Other Transactions
PrintReceipt
To print a receipt on PIN pads with built in printers.
Implementation Notes:
1. Currently available on the PAX A920Pro, PAX A920MAX, PAX A800, Ingenico DX4000, Ingenico DX8000, Ingenico Move/3500, Ingenico Move/5000 devices.
2. Each <LineN> should be preceded with a period character to preserve whitespace. Include a line containing only a period to print a blank line.
3. Responds with <CmdStatus> Success if receipt successfuly prints. Responds with <CmdStatus> Error if receipt fails to print.

* Optional

MerchantID
(String 1-24 chars)
	Required
Merchant identification assigned by processor.
	POSPackageID
(String 1-99 chars)
	Required
Include this tag with POS package name and version where the name and version are separated with a colon (“:”). Ex: EMVUSClient:1.26
	TranCode
(String 1-40 chars)
	Required
The type of transaction (eg. sale, return ,void, etc)
	SecureDevice
(String 1-40 chars)
	Required
Secure Device type. See Supported Devices for ValidSecureDeviceID values.
	SequenceNo
(String 1-20 chars)
	Required
Use the value returned for SequenceNo from the last response. If the SequenceNo is lost or for initial deployment of the PIN pad, the ECR/POS should attempt any transaction (using “0010010010” as a SequenceNo value) to re-sync. While this value is required, it’s unused in US EMV processing, and exists as a placeholder for future processing requirements.
	OperationMode
(String 1-4 chars)
	Conditional
Used to direct dsiEMVAndroid traffic to a specific NETePay Hosted environment. Values include “CERT” and “PROD”. Default value is “PROD”.
	PinPadIpAddress
(String 7-15 chars)
	Conditional
IP (v4) address of PIN pad on local network.
	PinPadIpPort
(Number 1-5 chars)
	Conditional
Defaults to 12000 for Ingenico TETRA pads. PAX devices listen on port 1235. Verifone devices listen on port 12345.

(String 1-24 chars)

	UserTrace

	TerminalID
(String 1-24 chars)
	Terminal ID data must be supplied in this tag only if required by the processor or merchant service provider; otherwise this tag should not be included. The POS system should store this value as a parameter to be included when required.

(String 1-10 chars)
	Indicates the source of the response:
“Server” = generated by DSI server
“Processor” = generated by processor
“Client” = generated by DSIClientX control
	DSIXReturnCode
(Number 6-6 chars)
	Six digit return code that identifies the error type. See Errors on error codes for possible values
	CmdStatus
(String 1-10 chars)
	Indicates the outcome of the command:
“Success” = command completed successfully
“Approved” = transaction approved by payment processor
“Declined” = transaction declined by payment processor
“Declined Offline” = chip transaction transaction declined offline
“Error” = error processing command. Check DSIXReturnCode and TextResponse for additional info on error. See Response Implementation Notes.
	TextResponse
(String 1-255 chars)
	For Successful responses, this field can contain an optional message from the server or processor.
	SequenceNo
(String 10-12 chars)
	The sequence number returned in all responses to be used on the next request. If the SequenceNo is lost or initial deployment of the PIN pad, the ECR/POS should attempt any transaction (using “0010010010” as a SequenceNo value) to re-sync.
	UserTrace
(String 0-40 chars)
	Echo of data supplied by the user system in the request; for use by the user system for internal tracking. May be null if no user system data was supplied in the request
	TranCode
(String 1-40 chars)
	TranCode value supplied in the transaction request.

	GetSignature
To obtain a digitized signature from a SecureDevice equipped with a digitizing pad
Implementation Notes:
1. The GetSignature transaction allows the integrator to optionally implement a paperless record system for drafts if the merchant is using a <SecureDevice> with signature capture capability. When a transaction response contains the <CVM> tag, the values can be one of the following:
“SIGN”
“PIN VERIFIED”
“PIN VERIFIED AND SIGN”
“NONE”
If the response value for <CVM> contains either “SIGN” or “PIN VERIFIED AND SIGN”, then perform a GetSignature transaction. The response from the GetSignature transaction should be saved with an electronic copy of the draft for that transaction to support a merchant in a charge back request. The GetSignature response data could be converted to a graphic format, such as .bmp and included in the stored draft image. In the absence of a CVM (when a non-EMV card is presented), a signature should be collected except when CardType is Debit.
2. The Signature response data is a string of up to 8192 characters and is returned as a series of “strokes” and within each stroke is a series of X,Y coordinate points. The first point of a stroke is always a “move to” position. All subsequent points within a stroke are “draw to” positions. The “end stroke point” is simply “#,#”. That would be followed by the first point of the next stroke. X and Y coordinates are separated by a comma and end with a colon.
3. An EMVPadReset should be performed prior to GetSignature to ensure the customer’s card is removed from the PIN pad.
