EBT Transactions
EBT Foodstamp EMVBalance
To process a foodstamp balance inquiry on an EBT card. Foodstamp EBT balance inquiries require the input of a PIN (Personal Identification Number) by the cardholder into a PIN pad device which encrypts the PIN number for secure transmission to the bank’s computer for validation.
Implementation Notes:
1. <TranCode> should be set to EMVBalance
2. <TranType> should be set to EBT
3. <CardType> should be set to Foodstamp
4. <Amount:Purchase> should be set to 0.00
5. If processing with WorldPay Core or Fiserv RapidConnect, EMV EBT is supported. In which case exclude <TranType> to support EMV EBT and MSR EBT cards.
Sample Request
* Required
* Optional
<?xml version="1.0"?>
  <TStream>
    <Transaction>
        <MerchantID>DEMODCHAL2GP</MerchantID>
        <POSPackageID>AndroidApp:1.02</POSPackageID>
        <OperationMode>CERT</OperationMode>
        <SecureDevice>EMV_A920PRO_DATACAP_E2E</SecureDevice>
        <TranType>EBT</TranType>
        <CardType>Foodstamp</CardType>
        <TranCode>EMVBalance</TranCode>
        <InvoiceNo>260119</InvoiceNo>
        <RefNo>260119</RefNo>
        <Amount>
            <Purchase>0.00</Purchase>
        </Amount>
        <SequenceNo>0010010010</SequenceNo>
        <PinPadIpAddress>192.168.1.151</PinPadIpAddress>
        <PinPadIpPort>1235</PinPadIpPort>
    </Transaction>
</TStream>

MerchantID
(String 1-24 chars)
	Required
Merchant identification assigned by processor.
	POSPackageID
(String 1-99 chars)
	Required
Include this tag with POS package name and version where the name and version are separated with a colon (“:”). Ex: EMVUSClient:1.26
	TranType
(String 1-24 chars)
	Conditional
Used with the CardType to specify payment type for MSR EBT transaction.
	CardType
(String 1-24)
	Required
Include with value “Foodstamp” or “Cash”. Only CardType is included for EMV EBT transactions.
	TranCode
(String 1-40 chars)
	Required
The type of transaction (eg. sale, return)
	InvoiceNo
(Number 1-8 chars)
	Required
Invoice number - sequential receipt number, check number, or other unique transaction identifier created and supplied by POS system.
	RefNo
(String 1-50 chars)
	Required
Transaction reference number. Use the same data as InvoiceNo.
	Amount:Purchase
(Number 1-8 chars)
	Required
Total amount of eligible EBT items.
	SecureDevice
(String 1-40 chars)
	Required
Secure Device type. See Supported Devices for ValidSecureDeviceID values.
	SequenceNo
(String 1-20 chars)
	Required
Use the value returned for SequenceNo from the last response. If the SequenceNo is lost or for initial deployment of the PIN pad, the ECR/POS should attempt any transaction (using “0010010010” as a SequenceNo value) to re-sync.
	OperationMode
(String 1-4 chars)
	Conditional
used to direct transactions to CERT or PROD NETePay Hosted environment. Supported values are “CERT” and “PROD”. Default value is “PROD”.
	PinPadIpAddress
(String 7-15 chars)
	Conditional
IP (v4) address of PIN pad on local network.
	PinPadIpPort
(Number 1-5 chars)
	Conditional
Defaults to 12000 for Ingenico TETRA pads. PAX devices listen on port 1235. Verifone devices listen on port 12345.
	Account:AcctNo
(String 1-24 chars)
	When the optional Account tag is included with the value “Prompt”, the PIN pad will prompt the operator for manual input of account number and expiration date.
	OKAmount
(String 1-24 chars)
	When the optional OKAmount tag is included with the value “Disallow”, dsiEMVAndroid will not prompt the cardholder to approve the transaction amount. If the OKAmount tag is omitted, the PIN pad will ask the cardholder to approve the transaction amount by default.
	TerminalID
(String 1-24 chars)
	Terminal ID data must be supplied in this tag only if required by the processor or merchant service provider; otherwise this tag should not be included. The POS system should store this value as a parameter to be included when required.
	LaneID
(Number 1-8 chars)
	A unique LaneID (lane number) can be supplied to identify each POS/ECR in a single store location with multiple payment points
	OperatorID
(String 1-24 chars)
	Operator (clerk, server, etc.) associated with the transaction.
	UserTrace
(String 1-24 chars)
	A unique value created and supplied by POS system

ResponseOrigin
(String 1-10 chars)
	Indicates the source of the response:
“Client” = generated by dsiEMVAndroid application
“Server” = generated by Datacap server
“Processor” = generated by payment processor
	DSIXReturnCode
(Number 6-6 chars)
	Six digit return code that identifies the error type. See Errors on error codes for possible values.
	CmdStatus
(String 1-10 chars)
	Indicates the outcome of the command:
“Success” = command completed successfully
“Approved” = transaction approved by payment processor
“Declined” = transaction declined by payment processor
“Declined Offline” = chip transaction transaction declined offline
“Error” = error processing command. Check DSIXReturnCode and TextResponse for additional info on error.
	TextResponse
(String 1-255 chars)
	For Successful responses, this field can contain an optional message from the server or processor. For other response codes, see Errors on Error Codes for values.
	SequenceNo
(Number 1-20 chars)
	The sequence number returned in all responses to be used on the next request. If the SequenceNo is lost or initial deployment of the PIN pad, the ECR/POS should attempt any transaction (using “0010010010” as a SequenceNo value) to re-sync.
	UserTrace
(String 0-40 chars)
	Echo of data supplied by the user system in the request; for use by the user system for internal tracking. May be null if no user system data was supplied in the request
	MerchantID
(String 1-24 chars)
	The merchant ID supplied in the transaction request.
	AcctNo
(String 1-24 chars)
	The account number from the card used for a manually entered transaction – will be truncated.
	CardType
(String 3-24 chars)
	Card type used on transaction. Values include: AMEX, VISA, M/C, DCLB, DCVR, JCB, Debit, Foodstamp, Cash, OTHER, UNDETERMINED (returned on tokenized transactions).
	TranCode
(String 1-40 chars)
	TranCode value in response will indicate if transaction was processed as FSA.
	AuthCode
(String 1-24 chars)
	The authorization code returned by the processor for the approved transaction. The AuthCode value should be saved since it is required for VoidSale, Adjust, VoiceAuth and PreAuthCapture transactions.
	CaptureStatus
(String 1-24 chars)
	Indicates whether the host has captured the transaction. eg. “Captured”, “NotCaptured”
	RefNo
(String 1-50 chars)
	A reference returned by the processor. Should initially match InvoiceNo. The RefNo value returned by the processor should be saved since it is required for subsequent VoidSale, Adjust, VoiceAuth and PreAuthCapture transactions.
	InvoiceNo
(Number 1-8 chars)
	The invoice number supplied in the transaction request.
	OperatorID
(Number 1-10 chars)
	Operator (clerk, server, etc.) associated with the Transaction.
	Amount:Purchase
(Number 1-8 chars)
	The purchase amount requested in the transaction (to 2 decimal places).
	Amount:Authorize
(Number 1-8 chars)
	The amount actually authorized for the transaction. This amount could be less than the Purchase amount requested and the POS/ECR system should verify to determine if additional tendering is required to satisfy the total payment.
	Amount:Balance
(Number 1-8 chars)
	EBT balance amount formatted to 2 decimal places. Included in response if returned by payment processor.
	Amount:FoodBalance
(Number 1-8 chars)
	Foodstamp balance amount formatted to 2 decimal places. Included in response if returned by payment processor.
	Amount:CashBalance
(Number 1-8 chars)
	Cash balance amount formatted to 2 decimal places. Included in response if returned by payment processor.
	AcqRefData
(String 1-200 chars)
	Acquirer Reference Data returned by processor.
	ProcessData
(String 1-999 chars)
	optionally returned by processor. This value must be retained for use in a subsequent requests.
	EntryMethod
(String 1-40 chars)
	Transaction entry method.
	RecordNo
(String 1-256 chars)
	Returned when optional<RecordNo> tag used in a request. Only card present EBT transactions are supported.
	Date
(String 1-40 chars)
	Transaction date
	Time
(String 1-40 chars)
	Transaction time
	PayAPI_Id
(String 1-34 chars)
	The Pay API reference number to be used through Pay API for follow up transactions.
	PrintData:LineNNN
(String 1-41 chars)
	Receipt print lines. Each <LineN> tag should be printed on a new line. Each <LineN> tag begins with a “.” which should not be printed – empty lines (blank) have only “.” as data. When <PrintData> elements exist, they should always be printed as the draft for the transaction.

	eWIC Use and Implementation Notes
eWIC transactions are used to electronically process payments for the WIC (Women, Infants and Children) state benefits programs. Each state administering a WIC program defines the specific requirements for systems deployed in that state.
An eWIC system must regularly download an updated Approved Product List (APL) file that is used as part of the transaction process to identify WIC vs. non-WIC items. The APL includes Categories and Sub-Categories of eligible items that can be used in an eWIC sale. Universal Product Codes (UPC’s) and Price Look Up (PLU) codes may also be provided in the APL and are used to identify state agency approved items available in a store for eWIC redemption.
EWIC transactions are based on redeeming food quantities by specific food category. EWIC systems use the APL to determine what items in a sale are eligible for eWIC redemption at the time the eWIC tender is offered at an eWIC capable POS.
An eWIC purchase transaction is a combination of two transactions: a pre-purchase balance inquiry (BalancePreVal) and a purchase (Sale). When an eWIC card is presented, the eWIC POS system must first query (Balance) the state records for the eWIC cardholder account to determine what item categories to which the account may be entitled.
To process the eWIC redemption (Sale), the POS then ‘looks back’ into the items accumulated in the transaction and compares them to the APL to determine which are eligible for eWIC redemption. Then the eWIC system submits a redemption (Sale) request to determine which items will be accepted for the sale. The request contains a UPC or PLU, quantity and item price for each item to be processed in the eWIC Sale transaction.
Responses for eWIC Sale and Voucher transactions may include item data with changes that differ from the request. The eWIC system must be able to adjust an eWIC redemption (Sale or Voucher) based on the response for each item since items may be denied or reduced in quantity and/or price.
eWIC responses include grouped information on each of the individual requested items in the following tags:
Tag
	Description
	UPCItem#
	UPC for item #
	PLUItem#
	PLU for item #
	ItemPrice#
	Actual price allowed for Item #
	ItemQty#
	Actual quantity allowed for Item #
	ItemStatus
	Status of Item # (see below)
	The eWIC system must evaluate the ItemStatus values for each Item#. The following chart is a guideline outlining the actions to be taken on each Item#:
Action
	Description
	Approved
	Item approved at price and quantity requested.
	Approved for lower qty
	Item approved for a lesser quantity than requested. POS should adjust eWIC sale detail as needed. POS should allow inclusion of excess item quantities in sale for alternate payment at cardholder’s discretion.
	Approved for lower price
	Item approved for a lesser price than requested. POS should adjust eWIC sale price detail as needed
	Denied
	Item not allowed for eWIC sale. POS should allow inclusion of items in sale for alternate payment at cardholder’s discretion.
	Denied CAT not prescribed

	Denied insufficient units

	Denied UPC/PLU not prescribed

	BIN Lists and eWIC Terminal IDs
Up to the first 8 digits of an eWIC card number are known as the Issuer Identification Number (IIN), also known as bank identification number (BIN). For eWIC cards, these are defined by the state that issues cards to the cardholders.
An eWIC system must verify that a card to be used for an eWIC transaction is allowed for the state by checking the BIN before submitting transactions for processing. A terminal ID assigned by the processing service provider may also need to be associated with the BIN. The responsible state agency is the source for the applicable BIN ranges supported; the processing service provider is the source for the Terminal IDs.
NETePay eWIC transaction requests include a tag, EWICBins, where the allowed card BIN ranges and Terminal IDs are supplied for verification by the processing service provider. The EWICBins tag must be included in every eWIC transaction request.
The format of the data for EWICBins tag is processor dependent as follows:
