Vertical Specific Features
Parking IN/OUT
To help POS parking systems, dsiEMVAndroid supports a method to uniquely identify a customer’s payment card, allowing the POS to determine when the customer is entering and exiting a parking lot or garage. Datacap’s control will return the CustomerData tag, a short-term local identifier of the presented card. This value is not to be stored for long-term use, please see CardHolderID
On the Way In
The parking system performs a CollectCardData request and the customer presents their card. This reads the presented card, and returns the CustomerData tag. All fields should be stored by the POS, along with the CustomerData field, which will be used to match the customer’s card on the way out.
The placeholder amount for the CollectCardData request should not be $0.00. The customer will not be charged for the request amount.

On the Way Out
The parking system performs an additional CollectCardData request and the payment card is presented again. This will again return the CustomerData tag. The CustomerData value can be checked against the previously collected CustomerData to confirm that the card matches, calculate the time difference and determine the total cost.
Once matched, the parking system can use the UseLastCardID field again in an EMVSale request, to charge the Customer for the calculated parking cost without the card being presented again.

Parking In/Out Implementation Notes
1. Match the value returned in the <CustomerData> tag to reference the cardholder from the original PreAuth and determine the final amount owed by the cardholder.
2. Use the value returned in the <Last4> as the value for <UseLastCardID>.
3. The payment terminal should be configured for “credit only” with no CVM to avoid PIN prompting.
4. Use the <OKAmount>Supress</OKAmount> to avoid prompting for amount acknowledgement
5. The parking system should reject any CustomerData values that already exist in their pending data store. This would mean the same payment card is being used to identify a second vehicle, and the card no longer serves as a unique identifier.
Line-Item Display
Support for Line-Item display/Line-Item detail was added in dsiEMVAndroid 2.09.
Line-Item Detail allows the pin pad to display product names or descriptions as provided by the Point of Sale.
To use Line-Item Display, the tag <UseLineItemDisplay>Allow<UseLineItemDisplay> must be supplied with the supported forms package on the device.
There are 3 new interface functions specific to line-item display, AddLineItem(), RemoveLineItem(), and ClearLineItems()

The following are some typical use cases for Lodging:
Check-in
A Check-In transaction authorizes a sale purchased with a credit card. The Check-In transaction utilizes the <PreAuth> or <EMVPreAuth> transaction if the amount is not meant to be settled. However, if the amount should be settled (e.g., if there is an advance deposit or fee to be charged) this can be done with a Sale or EMVSale. Also, if all that is needed at check in is to validate the card, a <ZeroAuth> or <EMVZeroAuth> can be used.
Check-out
A Check-Out transaction closes out a Check-In transaction and adds the transaction to the current batch. A Check-Out transaction utilizes the <PreAuthCaptureByRecordNo> transaction if the Check-In transaction was a PreAuth or EMVPreAuth and has not yet been added to a batch. Otherwise, a new Sale or EMVSale can be run. The Check-Out amount can be edited with an <AdjustByRecordNo> transaction. The transaction request includes the <AuthCode>, <RefNo>, <AcqRefData> and <RecordNo> from the Check-In transaction.
Incremental Authorization
Use the <IncrementalAuthByRecordNo> transaction to add to the authorized amount on a credit card. Incremental authorization in the lodging industry is typically used for additional duration and extra charges added to a customer’s stay.
* Single Stay—To charge for a Single Stay use the Sale or EMVSale transaction. This will authorize the associated amount and add it to the current batch. If a batch does not exist, this transaction creates one.
Note: The duration for a Single Stay defaults to one day.
* Advance Deposit—To run an Advance Deposit, use the Sale or EMVSale transaction. The merchant must ensure that the transaction amount does not exceed the total price of the reserved services or activity. The cardholder must be informed of the total price of the services or activity, the advance deposit amount, the advance deposit confirmation code, and the cancellation terms.
* Additional Charge—To include an additional charge, use the AdjustByRecordNo transaction to alter the original transaction.
* No Show—To charge penalties for a No Show, use a Sale transaction.
Note: The check in date is the initial authorization date and the stay duration is one day.
NationsBenefits Cards
Starting in dsiEMVAndroid 2.02, dsiEMVAndroid supports Nations Benefit Cards (swiped MC starting with 531445).
NationsBenefits offers a comprehensive flex benefit card solution that enables health plans to provide their members with a convenient way to pay for eligible items and approved services. Members receive a personalized debit card that gives them quick and easy access to funds that are loaded on a monthly, quarterly, or annual basis.
In order to support Nations Benefits cards, an ISV needs to collect the card data, but then determine eligibility prior to processing the payment. For this reason, dsiEMVAndroid needs to return the card data to the POS without processing.
Send a list of BINs in the <ReturnIfBinIn> tag. BINs should be comma separated.
Example - <ReturnIfBinIn>531445</ReturnIfBinIn> OR <ReturnIfBinIn>531445,532440,533552</ReturnIfBinIn>
During any EMV transaction that collects card data (EMVSale, EMVReturn, etc.), dsiEMVAndroid will determine if the card collected matches the BIN provided in the <ReturnIfBinIn> tag. If it does, then card data is returned similar to the data returned for Card Data Collection.
This feature can be used to return card data for any specific BIN or range of BINs.

Fleet card prompting (Driver ID, Odometer, etc.) is supported for PAX devices (attended and unattended).

Google Smart Tap for Value Added Services
Google Smart Tap for acceptance of tickets and passes using the Google Smart Tap Wallet on PAX pads was added in dsiEMVAndroid version 2.09.
1. The point of sale needs to request a Public Cerficate by using the TranCode GetSmartTapPublicKeyCertificate. The <PublicKeyCertificate> will be returned to the point of sale in the response. The point of sale must save the PublicKeyCertificate.
2. The point of sale generates a random 256-bit AES key.
3. The Google Smart Tap “Redemption Issuer” (the business that is accepting the Google Smart Tap) should have a Smart Tap Authentication Key. Datacap refers to this as the LongTermPrivateKey.
4. The point of sale’s AES key is then encrypted using the <PublicKeyCertificate>
5. You need to push the AES encrypted “Long Term Private Key” to the PAX device by using the SetSmartTapLongTermPrivateKey TranCode. You need to send the LongTermPrivateKeyEncryptedWithAES and the AesKeyEncryptedWithPubCert in your SetSmartTapLongTermPrivateKey TranCode.
6. After the key is set in the pad, you can now use the pad to accept Google Smart Tap passes and tickets using the GetVASData TranCode.
7. Decrypt the VASdata that was returned in the response to the GetVASData using the point of sale’s AES key
