Response Implementation Notes
Response Handling
If CmdStatus response is
	 
	Approved
	Transaction was successfully processed and cardholder account was charged
	Declined
	Transaction was successfully processed but cardholder account was declined and not charged– an alternative tender should be processed.
	Error
	Transaction was not successfully processed due to some error – See error codes.  Transaction should be retried or alternative tender attempted.
	Success
	Admin type request was successfully completed
	1. The TextResponse, CmdStatus, DSIXReturnCode and ResponseOrigin must be displayed for operator.
2. The order of the tags may differ from the presentation above and depends on the XML parser used..
Receipt Printing
Merchant processing agreements require that a separate receipt document must be printed in addition to the POS receipt.  Two copies must be printed; one for presentation to the cardholder (a separate receipt from the usual receipt) and one for retention by the merchant.  A PrintData stream is included in the dsiEMVAndroid response that can be used to print the receipt document(s).  Before printing the contents of the PrintData stream, the POS system should print the merchant name, address and phone number as separate lines at the top of the receipt.   After printing the contents of the PrintData stream, the POS must add the phrase “Cardholder Copy” as the last line of the first receipt copy and the phrase “Merchant Copy” as the last line of the second receipt copy. Cardholder Copy
Note: If the cardholder is prompted to enter a PIN during a transaction and does so successfully, the receipt copy will indicate that the cardholder was verified by PIN entry.  In the event that a requested PIN entry is unsuccessful, the receipt copy will include a signature line for the cardholder to sign.  The merchant should have the cardholder sign the Merchant Copy of the receipt and retain for their records.
As an alternative to using the <PrintData> stream, a software developer can create their own receipt documents from tag data included in the dsiEMVAndroid transaction response.  Two identical copies of the receipt document should be printed  with the exception that the POS must add the phrase “Cardholder Copy” as the last line of the first receipt copy and the phrase “Merchant Copy” as the last line of the second receipt copy.

	If CmdStatus = “Declined” and TextResponse = “Declined Offline.” then the POS should capture any PrintLine data returned in the response for printing a receipt if the cardholder requests.  Since the contents of the response for an offline decline will not necessarily contain the usual receipt tags, the POS system must utilize the <PrintData> lines.
