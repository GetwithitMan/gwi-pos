	ProcessTransaction
To process EMV credit and debit transactions using an attached SecureDevice to the NETePay Hosted platform.
Element
	Definition
	Library
	dsiEMVAndroid.aar
	Syntax
	String ProcessTransaction (String xmlRequest)
	Arguments
	String xmlRequest – An XML formatted string containing the transaction request. See EMV and Admin transactions for available XML commands, formats and usage.
	Returns
	XML formatted string response of type RStream. See XML Responses.
	Note
	dsiEMVAndroid provides the POS the ability to process payments without handling sensitive cardholder data (i.e. account number, expiration date, CVV/CID, PIN). dsiEMVAndroid directly controls the card account input device, such as an MSR, PIN pad, contactless reader, etc, facilitating an ‘out of scope’ position from a PCI-DSC (Payment Card Industry Data Security Council) PA-DSS (Payment Application – Data Security Standard) perspective for a POS application.

	CancelRequest
Attempts to cancel a previously initiated transaction for a device without a cancel button.
Element
	Definition
	Library
	dsiEMVAndroid.aar
	Syntax
	void CancelRequest ()
	Arguments
	None
	Returns
	None. If successful, the original ProcessTransaction will return a “Transacation Not Complete” message indicating the transaction as canceled by the user.
	Note
	A transaction can only be canceled until the point of a successful card read. Unsuccesful cancel attempts are ignored. Successful attempts will return from the ProcessTransaction method already in progress.

	EstablishBluetoothConnection
Establishes a Bluetooth connection to the Bluetooth device provided.
Element
	Definition
	Library
	dsiEMVAndroid.aar
	Syntax
	String EstablishBluetoothConnection (String bluetoothDeviceName)
	Arguments
	String bluetoothDeviceName – Bluetooth device to connect to. Use the identifier returned by Bluetooth device during BLE discovery.
	Returns
	XML formatted string response of type RStream containing whether connection was successful.
	Note
	The ProcessTransaction function implicitly calls EstablishBluetoothConnection if the supplied Bluetooth device is not already connected. Use EstablishBluetoothConnection prior to ProcessTransaction to avoid waiting for a Bluetooth connection while processing an active transaction (and potentially delaying customer interraction).

	Supported EMV Transactions and Operations

	EMVPadReset
	This command must be performed after every monetary transaction (Sale, VoidSale, Return, VoidReturn, VoiceAuth, ZeroAuth, etc) to assure that no card remains in the PIN pad chip reader. If no card is in the reader, then a response should be returned in 2-3 seconds. If there’s a card in the reader, the PIN pad displays ‘Remove Card’ and waits for the card to be removed before returning a response.

	EBT
Operation
	Definition
	EBT Foodstamp EMVSale
	Process a Foodstamp payment on an EBT card. Foodstamp EBT sales require the input of a PIN (Personal Identification Number) by the cardholder into a PIN pad device which encrypts the PIN number for secure transmission to the bank’s computer for validation.
	EBT Foodstamp EMVReturn
	Process a Foodstamp return (refund) on an EBT card. Foodstamp EBT returns require the input of a PIN (Personal Identification Number) by the cardholder into a PIN pad device which encrypts the PIN number for secure transmission to the bank’s computer for validation.
	EBT Foodstamp EMVBalance
	Process a foodstamp balance inquiry on an EBT card. Foodstamp EBT balance inquiries require the input of a PIN (Personal Identification Number) by the cardholder into a PIN pad device which encrypts the PIN number for secure transmission to the bank’s computer for validation.
	EBT Foodstamp EMVVoucher
	Process a Foodstamp voucher payment. The AcctNo, ExpDate, AuthCode and VoucherNo data must be supplied from the printed voucher.
	EBT Cash EMVSale
	Process a cash payment on an EBT card. Cash EBT sales require the input of a PIN (Personal Identification Number) by the cardholder into a PIN pad device which encrypts the PIN number for secure transmission to the bank’s computer for validation.
	EBT Cash EMVBalance
	Process a cash balance inquiry on an EBT card. EBT cash balance inquiries require the input of a PIN (Personal Identification Number) by the cardholder into a PIN pad device which encrypts the PIN number for secure transmission to the bank’s computer for validation.

	Additional Use Cases
Operation
	Definition
	Payment Fee Suite
	dsiEMVAndroid supports Datacap’s Payment Fee Suite (Compliant Surcharge with Lookup, Dual or multi-pricing, and Convenience Fee / Split fee funding)
	CardholderID
	CardholderID allows ISVs to track unique cardholders across their entire merchant base for data aggregation and proactive marketing.
	SuggestiveTipPrompt
	dsiEMVAndroid supports Sugggestive Tip Prompting on select supported PIN pads.
	Custom Amount Format
	On PAX Android devices, dsiEMVAndroid supports the <CustomAmountFormat> field, which allows integrators to display a custom message to the cardholder on the “collect card data” form screen.
	Custom Forms
	dsiEMVAndroid supports customs forms, so integrators can develop/design their own forms packages on PAX Android devices.
	Parking IN/OUT
	dsiEMVAndroid PAX Android devices support a method to uniquely identify a customer’s payment card, allowing the POS to determine when the customer is entering and exiting a parking lot or garage. Datacap’s control combines and hashes the unencrypted parts of the payment card, and provides them back to the POS as a base-64 encoded string.
	Stored Credential On File
	Ideal for situations of recurring payments and installment payments. By using COF, the merchant and POS avoid storing card data for recurring transactions, and maintain compliance with the card brands.
	Important: Support for any particular transaction code depends on the processing host’s support of that transaction.
