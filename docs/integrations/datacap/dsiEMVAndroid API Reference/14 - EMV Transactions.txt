EMV Transactions
EMVPadReset
To reset and ready the EMV PIN pad device for a new transaction.
Implementation Notes:
1. <TranCode> should be set to EMVPadReset
2. This command should be performed at the end of any card related transaction (Sale, VoidSale, Return, VoidReturn, VoiceAuth and ZeroAuth) to assure that the user is prompted to remove their card. If no EMV chip card is inserted in the reader, then an EMVPadReset will return a successful response in one second. If there’s an EMV chip card in the reader, then the EMVPadReset command will wait up to 15 seconds for removal. If the card is removed within that timeframe, then a successful response is returned when the card is removed. If the chip card remains in the reader after 15 seconds, an error response is returned “TRANSACTION NOT COMPLETE - Card Not Removed”.

	EMVSale
To process a payment using EMV capable equipment. Credit and Debit transactions with chip, MSR and manually input account information are supported.
Functional Option Tags for EMVSale:
1. Token Request - (Tags: <RecordNo> and <Frequency>) – Used to request an optional unique transaction identifier (token) to be assigned by the payment processor and returned in the transaction response. This token value is supplied to a subsequent AdjustByRecordNo transaction request to alter the amounts on the original transaction without possession of the card (e.g. for tip entry). Note that not all processors support token issuance or subsequent use and that only cards that don’t require a PIN entry during the EMVSale will be able to request a token.
2. FSA Sale – If the BIN of a swiped credit transaction matches one of the BINs provided in the <FSABins> tag, the transaction will be switched to an FSA sale transaction using the provided FSA fields (FSATotalAmount, FSAPrescription, FSAVision, FSADental, FSAClinical).
3. PartialAuth acceptance is required for select processors. In the case of a partial approval, the response <Authorize> amount will be less than the requested <Purchase> amount. The POS will need to perform a second transaction to collect the balance. Alternatively, the POS can issue a void, sending in the previously authorized amount in the <Purchase> field.
4. For Fleet fields, in each <LineItem> contained inside the <PetroleumItems> tag, name numbers 1-9 are supported for a total of 9 <LineItem> items. eg, <LineItem1>, <ItemCode1> ... </LineItem1> <LineItem2> <ItemCode2> ... </LineItem2>. Note: Fleet is only supported on Fiserv Rapid Connect today.
Implementation Notes:
1. The <Duplicate> tag is not supported by all EMV processing providers. In the event that a duplicate response is received for a request (<TextResponse> = “AP_DUPE”), using a value of “Override” for the <Duplicate> tag on a subsequent attempt will force the processing host to skip duplicate checking and therefore approve what appears to be a duplicate transaction. This is useful in cases where multiple transactions of the same amount to the same card within a short time period are necessary. Do not send the “Override” value unless the previous transaction attempt resulted in a duplicate response otherwise the processing host will return <TextResponse> = “NO_DUPE_FOUND”.
2. To request a token, set <RecordNo> to RecordNumberRequested

	EMVSaleWithSplit
To process a payment using EMV capable equipment along with a separate, second transaction for a “split fee” amount. Credit and Debit transactions with chip, MSR and manually input account information are supported.
Implementation Notes
1. EMVSaleWithSplit requires NETePay Hosted.
2. The <SplitMerchantID1> is the Hosted Merchant ID value that should be used to process the split fee payment.
3. <SplitFee1> is the amount of the split fee payment.
4. <CustomAmountFormat> is a field that contains a display message to be shown to the cardholder on a PAX PIN pad showing the amount of the EMVSale and the amount of the split fee. The value of the <CustomAmountFormat> tag maps to a text box contained in the latest PAX forms package. dsiEMVAndroid checks the PAX forms package version to know if the variable exists before populating the variable for amount confirmation and when displaying the form to collect card data (Please Insert, Swipe or Tap Card). An example of the CustomAmountFormat sent on an EMVSaleWithSplit is <CustomAmountFormat>Payment Amount: $#PAYMENT##N#Fee Amount: $#FEE1#</CustomAmountFormat>.
5. The print data returned by dsiEMVAndroid is the EMV certified receipt with all required EMV fields pertaining to the primary transaction. It is up to the integrator to augment the receipt they print for the customer to include the split fee and display it as the integrator sees fit.

	EMVPreAuth
To preauthorize a credit card which must be subsequently captured for settlement. Credit cards only are supported
Functional Option Tags for EMVPreAuth:
1. Gratuity Entry - (Tag: <Gratuity>) – Allows a cardholder to specify a tip for a transaction in addition to a base transaction amount. Usually used by restaurant and service merchants.
2. Token Request - (Tags: <RecordNo> and <Frequency>) – Used to request a unique transaction identifier (recurring token) to be assigned by the payment processor and returned in the transaction response. This token value is supplied to a subsequent PreAuthCaptureByRecordNo transaction request to alter the amounts on the original transaction without possession of the card (e.g. for tip entry). Note that not all processors support token issuance or subsequent use and that only cards that don’t require a PIN entry during the EMVSale will be able to request a token.
Implementation Notes:
1. <CardType> should be set to Credit
2. To request a token, set <RecordNo> to RecordNumberRequested
3. The <Duplicate> tag is not supported by all EMV processing providers. In the event that a duplicate response is received for a request (TextResponse=“AP_DUPE”), using a value of “Override” for the <Duplicate> tag on a subsequent attempt will force the processing host to skip duplicate checking and therefore approve what appears to be a duplicate transaction. This is useful in cases where multiple transactions of the same amount to the same card within a short time period are necessary. Do not send the “Override” value unless the previous transaction attempt resulted in a duplicate response otherwise the processing host will return TextResponse=“NO_DUPE_FOUND”.
4. Manual card account entry with the <AcctNo> tag should only be used if the PIN pad insertion reader fails to get card account information on the first transaction attempt. The POS logic should allow the transaction to be run again with the <AcctNo> tag included to force a manual entry of card account information.
5. For a chip card, the EMV tag data is returned in the <ProcessData> field. It is required to send this data on the PreAuthCaptureByRecordNo.

	EMVReturn
To process a return (refund) using EMV capable equipment. Credit and Debit transactions with chip, MSR and manually input account information are supported.
Functional Options for EMVReturn:
1. Token Request - (Tags: <RecordNo> and <Frequency>) – Used to request an optional unique transaction identifier (token) to be assigned by the payment processor and returned in the transaction response. This token value is supplied to a subsequent AdjustByRecordNo transaction request to alter the amounts on the original transaction without possession of the card (e.g. for tip entry). Note that not all processors support token issuance or subsequent use and that only cards that don’t require a PIN entry during the EMVReturn will be able to request a token.
Implementation Notes:
1. The Duplicate tag is not supported by all EMV processing providers. In the event that a duplicate response is received for a request (TextResponse=“AP_DUPE”), using a value of “Override” for the Duplicate tag on a subsequent attempt will force the processing host to skip duplicate checking and therefore approve what appears to be a duplicate transaction. This is useful in cases where multiple transactions of the same amount to the same card within a short time period are necessary. Do not send the “Override” value unless the previous transaction attempt resulted in a duplicate response otherwise the processing host will return TextResponse=“NO_DUPE_FOUND”.
2. To request a token, set <RecordNo> to RecordNumberRequested
3. To utilize Linked Returns, enable the Linked Returns setting in PSCS. To link a return to the sale or capture, save the PayAPI_Id value from the initial response and include it as the RefNo value in the request. The <ReturnAmountRemaining> tag is returned in the response with the remaining amount from the original sale or capture. The <ReturnAmountRemaining> value is the <Authorize> amount returned in the original sale or capture. A Linked Return of the full amount will result in 0.00 being returned.
