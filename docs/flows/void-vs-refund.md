# Flow: Void vs. Refund

> **When to read this:** Before changing any feature listed in §8 Dependencies. If your change touches this journey, read this doc first.

---

## 1. Purpose

**Trigger:** A manager needs to reverse a payment — either to cancel a mistake (void) or to return money to a customer after settlement (refund).

**Why it matters:** Money integrity. Voids and refunds are irreversible financial operations. The wrong path — voiding a settled payment or refunding an unsettled one — corrupts the payment ledger, incorrectly reverses tip allocations, and produces unrecoverable reporting errors. The decision gate (`Payment.settledAt`) is the single authoritative branch point and must never be bypassed.

**Scope:** `gwi-pos` (void-payment and refund-payment API routes, VoidPaymentModal, ClosedOrderActionsModal, Datacap client, AuditLog), `gwi-android-register` (syncs voided/refunded payment status; does not initiate voids or refunds).

---

## 2. Preconditions

| Precondition | Detail |
|-------------|--------|
| Feature flags / settings | Datacap reader must be configured and online for refund path (EMV return requires network); void path is local DB only |
| Hardware required | Active `PaymentReader` with `communicationMode = 'local'` required for refund path; not required for void |
| Permissions required | `manager.void_payments` (Critical/High) gates BOTH void and refund; standard employees cannot perform either action |
| Online / offline state | Void path works fully offline (local DB only); refund path requires NUC → Datacap reader LAN connectivity |
| Prior state | A `Payment` record in `status = 'completed'` must exist; payment must not already be `voided` or fully `refunded` |

## 3. Sequence (Happy Path)

### Decision Gate — Which Path?

```
1. [CLIENT]       Manager opens order detail panel → payment history section
                  OR opens ClosedOrderActionsModal from /settings/orders/closed
2. [CLIENT]       VoidPaymentModal auto-selects path:
                  payment.settledAt === null → VOID PATH
                  payment.settledAt is set   → REFUND PATH
```

---

### Path A — Void (Pre-Settlement)

```
3A. [CLIENT]      VoidPaymentModal shows "Void Payment" UI; manager enters void reason + Manager PIN
4A. [API]         POST /api/orders/[id]/void-payment
                  requirePermission(manager.void_payments) → 403 if not authorized
5A. [GUARD]       Checks: payment.status !== 'voided' → 409 if already voided
                  payment.settledAt must be null → 403 if settled (wrong path)
6A. [DB]          Prisma $transaction:
                  Payment: status = 'voided', voidedAt = NOW(), voidedBy = managerId, voidReason
                  If no other active payments on order: Order.status = 'voided'
                  AuditLog created: { action: 'payment_voided', employeeId, ip, userAgent, context }
7A. [TIP REVERSE] handleTipChargeback() called fire-and-forget:
                  Policy evaluation (BUSINESS_ABSORBS | EMPLOYEE_CHARGEBACK | MANAGER_APPROVAL)
                  If EMPLOYEE_CHARGEBACK: TipLedgerEntry DEBIT posted to reverse employee tip credit
                  If MANAGER_APPROVAL: TipDebt record created for manager review
8A. [EVENTS]      emitOrderEvent('PAYMENT_VOIDED', { paymentId, reason })
9A. [SNAPSHOT]    OrderSnapshot rebuilt: paidAmountCents reduced, isClosed may flip to false
10A. [BROADCAST]  dispatchPaymentProcessed() → emitToLocation('payment:processed', { orderId, paymentId, status: 'voided' })
                  emitToLocation('order:totals_updated', { orderId, totals })
11A. [SIDE EFFECTS] Android syncs updated payment.status via standard payment sync path;
                   No Datacap call is made — void is entirely local DB
                   No receipt generated by default
```

---

### Path B — Refund (Post-Settlement)

```
3B. [CLIENT]      VoidPaymentModal shows "Refund Payment" UI; manager enters:
                  refundAmount (full or partial), refundReason, notes, Manager PIN, readerId
4B. [API]         POST /api/orders/[id]/refund-payment
                  requirePermission(manager.void_payments) → 403 if not authorized
5B. [GUARD]       Validates refundAmount > 0 → 400 if zero
                  Validates refundAmount <= payment.amount → 400 if over-refund
                  payment.status !== 'voided' → 403 if already voided
                  payment.status !== 'refunded' (full refund) → 403 if fully refunded
6B. [DATACAP]     If paymentMethod is 'credit' or 'debit' AND readerId provided AND datacapRecordNo set:
                  client.emvReturn(readerId, { recordNo: payment.datacapRecordNo,
                    invoiceNo: order.orderNumber, amount: refundAmount, cardPresent: false })
                  Datacap EMVReturn call (CreditByRecordNo / ReturnByRecord)
7B. [GUARD]       If Datacap response cmdStatus !== 'Approved' → HTTP 422 returned;
                  DB is NOT written; payment remains 'completed'
8B. [DB]          On Datacap approval, Prisma $transaction:
                  Payment: status = 'refunded' (full) or remains 'completed' (partial)
                  Payment: refundedAt = NOW(), refundedAmount += refundAmount, refundReason
                  RefundLog created: { orderId, paymentId, employeeId, refundAmount, originalAmount,
                    refundReason, notes, datacapRecordNo, datacapRefNo, approvedById }
                  AuditLog created: { action: 'payment_refunded', employeeId, ip, userAgent }
9B. [TIP ADJUST]  Tip adjustment logic:
                  Full refund: tip zeroed — TipLedgerEntry DEBIT reverses full tip credit
                  Partial refund: tip recalculated proportionally — delta DEBIT entry posted
                  All tip reversals via handleTipChargeback() fire-and-forget
10B. [EVENTS]     emitOrderEvent('PAYMENT_VOIDED', { paymentId, reason }) — reuses PAYMENT_VOIDED event type
11B. [SNAPSHOT]   OrderSnapshot rebuilt
12B. [BROADCAST]  emitToLocation('payment:processed', { orderId, paymentId, status: 'refunded' })
                  emitToLocation('order:totals_updated', { orderId, totals })
13B. [SIDE EFFECTS] RefundLog.receiptPrinted set when receipt printed;
                   Android syncs payment.status and refundedAmount via standard sync;
                   Reports: refunded amounts deducted from daily/shift revenue totals
```

---

### Item-Level Comp-Void (Separate Subsystem — Not a Payment Operation)

```
NOTE: POST /api/orders/[id]/comp-void is a DIFFERENT subsystem. It voids or comps
individual line items from an open order BEFORE payment. It does not touch Payment records,
does not create RefundLog entries, and does not require Datacap.

Key distinctions:
- Comp-void targets OrderItem (not Payment)
- Used BEFORE payment, not after
- Requires manager.void_items (not manager.void_payments)
- Supports remote approval (6-digit code flow, Skill 121)
- Inventory: comps always deduct as waste; voids deduct only if wasMade = true
- Auto-closes order if all items voided/comped
- PUT /api/orders/[id]/comp-void undoes a void: OrderItem.status = 'active' restored

DO NOT conflate item-level comp-void with payment void or refund.
```

## 4. Events Emitted

| Event Name | Payload (key fields) | Emitter | Consumers | Ordering Constraint |
|------------|---------------------|---------|-----------|---------------------|
| `PAYMENT_VOIDED` | `{ paymentId, orderId, reason }` | POS API | Android, Backoffice | After DB write (both paths) |
| `payment:processed` | `{ orderId, paymentId, status }` | socket-dispatch.ts | All POS clients, Android | After DB commit |
| `order:totals_updated` | `{ orderId, totals }` | socket-dispatch.ts | All POS clients | After void changes order balance |

**Note:** Both void and refund paths reuse the `PAYMENT_VOIDED` order event type. There is no separate `PAYMENT_REFUNDED` event type in the 17-event schema.

---

## 5. State Changes

| Record | Fields Changed | When |
|--------|---------------|------|
| `Payment` | `status = 'voided'`, `voidedAt`, `voidedBy`, `voidReason` | Void path — DB commit |
| `Order` | `status = 'voided'` (if no other active payments) | Void path — DB commit |
| `Payment` | `status = 'refunded'` (full) or unchanged (partial), `refundedAt`, `refundedAmount`, `refundReason` | Refund path — after Datacap approval |
| `RefundLog` | Created: full refund context including `datacapRefNo` | Refund path only |
| `AuditLog` | Created: `action = 'payment_voided'` or `'payment_refunded'`, IP, userAgent | Both paths |
| `TipLedgerEntry` | DEBIT entry created to reverse tip (IMMUTABLE — new entry, not update) | Both paths — fire-and-forget |
| `TipDebt` | Created if chargeback policy = MANAGER_APPROVAL | Void path tip chargeback |
| `OrderSnapshot` | Rebuilt: `paidAmountCents` reduced, `isClosed` may change | After PAYMENT_VOIDED event |

**Snapshot rebuild points:** After `PAYMENT_VOIDED` event is emitted in both paths.

**Payment.status valid transitions:**
- `completed → voided` (void path only; payment.settledAt must be null)
- `completed → refunded` (full refund only; payment.settledAt must be set)
- `completed` stays `completed` for partial refund (RefundLog created but status unchanged)

No other transitions are valid.

## 6. Edge Cases

| Scenario | Behavior |
|----------|---------|
| **Voiding a settled payment** | API returns 403 — `payment.settledAt` is set, wrong path; use refund |
| **Refunding an unsettled payment** | API returns 403 — `payment.settledAt` is null, wrong path; use void |
| **Refunding a voided payment** | API returns 403 if `payment.status === 'voided'` |
| **Refunding a fully refunded payment** | API returns 403 if `payment.status === 'refunded'` |
| **Partial refund — second partial** | Allowed as long as cumulative `payment.refundedAmount` does not exceed `payment.amount`; each creates its own RefundLog |
| **Cash or house-account payment refund** | `isCardPayment` is false; Datacap call is skipped; RefundLog still created with `datacapRefNo = null` for record-keeping |
| **Void during SAF (offline capture)** | If payment captured offline (safStatus = SAF_PENDING_UPLOAD), void must occur BEFORE the SAF batch is forwarded (`SAF_ForwardAll`); once forwarded, payment is settled and void path is blocked |
| **Tip impact — full void** | `handleTipChargeback()` fires fire-and-forget; policy determines if employee ledger is debited or a TipDebt is created; if tip was never allocated, warning is logged but void does not fail |
| **Tip impact — full refund** | Full refund zeroes tip: DEBIT TipLedgerEntry for full tip amount posted to employee ledger |
| **Tip impact — partial refund** | Partial refund triggers proportional tip recalculation: delta DEBIT entry posted (e.g., refunding 50% of payment recalculates tip at 50%) |
| **Multiple payments on one order** | Void/refund targets a specific `paymentId`; if voiding one payment does not leave the order fully paid, order status may revert from 'closed' to 'open'; each payment is voided/refunded independently |
| **Chargeback vs. initiated refund** | System-initiated refund uses `POST /api/orders/[id]/refund-payment` (EMV return via Datacap). A card-network-initiated chargeback arrives externally; manager must void the payment manually and the chargeback is handled as a `BUSINESS_ABSORBS` or `MANAGER_APPROVAL` TipDebt, depending on policy |
| **Reader offline for refund** | If Datacap reader is unreachable, `emvReturn()` times out (60s local); API returns 422; DB is NOT written; manager must wait for reader to come back online or contact Datacap directly |
| **Permission denied** | If requesting employee does not have `manager.void_payments`: API returns 403 before any DB or Datacap interaction; client shows permission error |

## 7. Invariants (Never Break These)

- **[VR-1]** NEVER void a payment with `settledAt` set. Voiding settled payments corrupts the card network batch. Check `payment.settledAt === null` before calling the void route.
- **[VR-2]** NEVER refund an unsettled payment. Datacap EMV return against an unsettled charge will fail or produce a double-refund. Check `payment.settledAt !== null` before calling the refund route.
- **[VR-3]** NEVER write to the DB on refund until Datacap returns `cmdStatus === 'Approved'`. If the DB is written before Datacap approves, money can be refunded in the system but not at the card network.
- **[VR-4]** NEVER void without checking tip impact. `handleTipChargeback()` MUST be called after every payment void. If a tip was credited to an employee's ledger, it must be reversed via a DEBIT entry.
- **[VR-5]** Full refund zeroes tip. Partial refund recalculates tip proportionally. NEVER leave a tip standing on a fully refunded payment.
- **[VR-6]** TipLedgerEntry records are IMMUTABLE. Tip reversals post new DEBIT entries — never update or delete the original CREDIT entry.
- **[VR-7]** `manager.void_payments` is required for BOTH void and refund. There is no separate `manager.refunds` route gate — both actions share the same permission key.
- **[VR-8]** Item-level comp-void (POST /api/orders/[id]/comp-void) is NOT a payment reversal. Do not use it to reverse money. It modifies OrderItem records on open orders before payment and requires `manager.void_items`, not `manager.void_payments`.

If you break VR-3 (DB before Datacap approval): the only recovery is a manual reversal entry in RefundLog and a separate Datacap return call. Prevent this by keeping the Datacap call before the Prisma transaction in the refund route.
